### Task 3 (A) +scaffolding
blockedBy: [0]

**PURPOSE** — Every API integration caches responses to avoid redundant network calls and respect rate limits. SQLite is the cache backend. Setting it up once here means every API client just calls `cache.get()` / `cache.set()` without worrying about storage.

**WHAT TO DO**
1. Create `packages/shared/src/cache.ts`.
2. Import `better-sqlite3` and create/open `~/.sg-apis/cache.db` (create directory if missing).
3. Create table: `CREATE TABLE IF NOT EXISTS cache (key TEXT PRIMARY KEY, value TEXT NOT NULL, cached_at INTEGER NOT NULL, ttl_seconds INTEGER NOT NULL)`.
4. Implement `get(key: string): CachedValue | null` — return null if missing or if `Date.now()/1000 - cached_at > ttl_seconds`.
5. Implement `set(key: string, value: string, ttl: number): void` — upsert with `INSERT OR REPLACE`.
6. Implement `invalidate(pattern: string): number` — `DELETE FROM cache WHERE key LIKE ?`, return deleted count.
7. Implement `stats(): CacheStats` — return `{ entries: number, hits: number, misses: number, sizeBytes: number }`. Track hits/misses via in-memory counters.
8. Define TTL constants in `packages/shared/src/ttl.ts`:
   - `REALTIME = 30` — WHY: live data like taxi availability, stale after 30s
   - `NEAR_REALTIME = 300` — WHY: exchange rates update every few minutes, 5-min cache balances freshness vs. rate limits
   - `DAILY = 3600` — WHY: SingStat/URA data updates at most daily, 1-hour cache is safe
   - `STATIC = 86400` — WHY: planning area boundaries, coordinate conversions — data changes quarterly at most
   - `ARCHIVAL = 604800` — WHY: historical time series never change, cache for 7 days

**DONE WHEN**
- [ ] `cache.set("test", "value", 60)` followed by `cache.get("test")` returns `"value"`
- [ ] `cache.get("test")` returns `null` after TTL expiry (testable by setting TTL=0)
- [ ] `cache.invalidate("singstat:%")` deletes only SingStat-prefixed keys
- [ ] `cache.stats()` returns correct entry count
- [ ] TTL constants exported and documented with WHY comments

### Task 12 (B) +mcp-core
blockedBy: [1, 6]

**PURPOSE** — Exponential backoff with jitter prevents thundering herd problems when an API returns 429 or 503. Without jitter, all retries fire at the same instant and the API stays overloaded.

**WHAT TO DO**
1. Create `packages/shared/src/http-client.ts`.
2. Implement `httpGet<T>(url: string, options: HttpOptions): Promise<T>` wrapping `fetch()`.
3. `HttpOptions`: `{ headers?: Record<string, string>, timeout?: number, retries?: number, apiName: string }`.
4. Default timeout: `10000` — WHY: government APIs can be slow, but 10s is reasonable for a single request.
5. Default retries: `3` — WHY: covers transient failures without excessive delay.
6. Retry logic: on 429/500/502/503/504, wait `min(baseDelay * 2^attempt + jitter, maxDelay)` ms.
   - `baseDelay = 1000` — WHY: 1s base avoids hammering a recovering API.
   - `maxDelay = 15000` — WHY: 15s cap prevents unreasonable waits.
   - `jitter = random(0, 500)` — WHY: 500ms jitter window spreads retries.
7. Parse `Retry-After` header if present and use it as the wait time.
8. Integrate rate limiter: call `getRateLimiter(apiName).acquire()` before each request.

**DONE WHEN**
- [ ] 429 response triggers retry with exponential backoff
- [ ] `Retry-After: 5` header causes 5-second wait
- [ ] After max retries, throws `ApiError` with `retryable: true`
- [ ] Rate limiter is consulted before each request attempt

### Task 60 (B) +hardening
blockedBy: [8, 12]

**PURPOSE** — A circuit breaker prevents the server from repeatedly calling a broken API. After 3 consecutive failures, the circuit opens and returns a cached response or error immediately for 60 seconds.

**WHAT TO DO**
1. Create `packages/shared/src/circuit-breaker.ts`.
2. Implement `CircuitBreaker` class:
   - States: `CLOSED` (normal), `OPEN` (failing, skip calls), `HALF_OPEN` (test single call).
   - `failureThreshold = 3` — WHY: 3 failures is enough to detect a pattern, not too trigger-happy.
   - `resetTimeout = 60000` — WHY: 60 seconds gives the API time to recover.
   - `execute<T>(fn: () => Promise<T>): Promise<T>`:
     - CLOSED: execute fn, track failures. After threshold, transition to OPEN.
     - OPEN: throw `CircuitOpenError` immediately (do not call API). After resetTimeout, transition to HALF_OPEN.
     - HALF_OPEN: execute fn. If success, transition to CLOSED. If failure, transition to OPEN.
3. Integrate with `httpGet()`: wrap each API call in its API-specific circuit breaker.

**DONE WHEN**

- [ ] After 60 seconds, one test call is allowed (half-open)
- [ ] Successful test call closes the circuit (normal operation resumes)
- [ ] Circuit state is logged at info level on transitions
